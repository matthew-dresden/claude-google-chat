### Added

- **Foolproof onboarding: `cgc setup` (guided wizard) + `cgc doctor` (diagnostics).** Onboarding is now a single command. All gcloud, OAuth/ADC, and network calls go through injectable runner interfaces (`probes.py`), so the whole flow is unit-tested with fakes — no test touches the network, gcloud, real disk, or `sleep`.
  - **Shared error-mapping layer** (`errors.py`): `map_error` / `map_http_error` turn common Google API / transport failures into a concise, **actionable, secret-free** one-line message (never a raw traceback, request URL, or token). HTTP `401`/invalid-credentials → "re-authenticate: run 'cgc setup --reauth'"; `403` insufficient-scope → names the missing OAuth scope + re-auth; `403 PERMISSION_DENIED` → "you need a role on the project, or pick another project"; `404` → "space/thread/webhook not found — it may have been deleted; re-create it"; `429`/`5xx` → "transient — retry". Reused across setup/doctor (and available to chat/listener). `format_missing_scopes` is the single source of truth for the missing-scope wording.
  - **`cgc doctor`** — prints a RED/GREEN (`[PASS]`/`[FAIL]`) checklist of every prerequisite with the **exact fix command** per red line: gcloud installed? logged in? a project selected? Chat API enabled? OAuth/ADC credentials present & valid? token carries every required Chat scope? `webhook_url` configured & well-formed (token never echoed)? `space_id` configured? config file present? Each check is a small pure function over injectable probes returning `{name, ok, fix}`; **exits non-zero** when any required check fails (so it doubles as a health gate). Colour is suppressed off a TTY. Folds in the old trivial `cgc setup` behaviour by printing the config path.
  - **`cgc setup`** (replaces the old path-printing `setup`) — an **idempotent, resumable, verifying** wizard. Each step preflight-skips when already satisfied and re-running fixes only the gaps: detect gcloud (else print the install link + console deep-links for the manual project/API fallback); create or select a project; enable `chat.googleapis.com` and **poll for readiness** (active probe, not `sleep`; env-driven timeout `CGC_SETUP_ENABLE_TIMEOUT` / `CGC_SETUP_ENABLE_POLL_INTERVAL`, fail-fast on timeout); authenticate **ADC-first** (`gcloud auth application-default login` with the Chat + identity scopes, then probe whether the Chat API accepts that token — no OAuth client to create) with a **guided OAuth-client fallback** (exact console deep-links + client-secrets shape validation + `cgc auth login`); after either path **verify the token carries every required Chat scope** and re-auth if not (guards silent scope-drop); prompt for and validate the incoming-webhook URL shape (stored, token never echoed); and finally **verify end-to-end** with a real send + read-back round trip before declaring success. On any failure it prints the mapped actionable error naming which step to re-run. Flags: `--reauth` (just redo auth), `--dry-run` (show actions, change nothing), `--verify` (just run the round-trip check). The `gcloud`/auth/network/clock boundaries are injected via `Probes` so tests drive them with fakes.
  - **Plumbing**: `cgc setup` and `cgc doctor` are registered in `cli.py` (and surface in `--help` / completion); the new readiness-timeout tunables are read from the environment (no hardcoded values). New modules `errors.py`, `probes.py`, `doctor.py`, `setup.py` with real, injectable tests (`tests/test_errors.py`, `tests/test_probes.py`, `tests/test_doctor.py`, `tests/test_setup.py`, shared `tests/fakes.py`) covering: per-status error mapping; doctor RED/GREEN + per-line fixes + non-zero exit; setup idempotency, resume-after-failure, ADC-accepted vs ADC-rejected branches, scope-drop detection + re-auth trigger, `--dry-run` changing nothing, the readiness poll, and the round-trip gate failing setup when send/read fails. Documented in `README.md`, `docs/installation.md`, `docs/usage.md`, and `docs/configuration.md`. Line coverage remains at 100%.

### Removed

- **App-auth / standalone-bot half removed — the tool is now a single, session-bound path.** `claude-google-chat` is now exclusively the user-OAuth two-way integration: outbound status pings via the incoming **webhook** (`cgc chat send`) and inbound reading via the Chat REST API with **user OAuth** (`cgc listen`). The entire service-account "Chat app" side has been deleted, eliminating the prior "two ways to run this" split. Specifically removed:
  - **Commands `cgc serve` and `cgc bootstrap`** and their modules `serve.py` (the standalone responder) and `bootstrap.py` (service-account / space / Workspace-Events setup).
  - **The entire `terraform/` directory** (and `terraform/README.md`).
  - **Service-account (app) auth** in `auth.py` (`APP_SCOPES`, `load_app_credentials`, the service-account key loader) and in `chat.py` (`build_app_service`, `post_message_as_app`, `list_messages_as_app`). User OAuth (`CHAT_SCOPES`, `load_credentials`, `login`) is unchanged.
  - **Workspace Events + Pub/Sub** wiring (subscription creation, topic normalization, the events-service builder).
  - **App-auth config keys** `service_account_file`, `project_id`, `pubsub_topic`, `space_display_name`, and `owner_email`, plus their `CGC_*` env overrides and the completion key set. The remaining keys are unchanged (`webhook_url`, `space_id`, `oauth_client_file`, `token_file`, `trigger_prefix`, `poll_interval`, `listen_timeout`, `page_size`, `send_envelope`, `require_trigger`, `state_file`, `max_consecutive_errors`, `webhook_timeout`). `rawmessage.py` drops the now-unused `sender_email`/`thread_key` accessors (owner gating and threaded replies were serve-only); `sender_type`/`is_human_message` remain for the listener's loop-prevention.
  - **Docs** `docs/SETUP.md` (the app-auth runbook). README, `docs/installation.md`, `docs/usage.md`, `docs/architecture.md`, and `docs/configuration.md` were reframed around the single user-OAuth path: the "two auth paths" overview, the `serve`/`bootstrap` command docs, the Terraform/Pub/Sub/Workspace-Events references, and the phone-notifications section were removed, and the architecture module list now reflects the remaining modules. Tests targeting the removed surface were deleted; the shared fixtures and the remaining suite were updated, keeping line coverage at 100%.

### Added

- **Session layer** — a named, durable binding (on top of the thread primitives) between a working context and one or more Chat threads in a shared space, so multiple Claude Code instances can share one space. New modules `sessions.py` (pure: `Session`/`SessionThread`, the `SessionRegistry` protocol with `FileSessionRegistry` (JSON `0600`; malformed → fail fast, never silently discarded) and `InMemorySessionRegistry`, deterministic name derivation/sanitization, idempotent `upsert`/`add_thread`/`remove` with dispatcher election + promotion, the `NAME:` prefix split, and the pure `route_message` decision), `sessionops.py` (connect/list/disconnect orchestration with injectable registry + thread-sender + clock + git-context), and `sessionrouter.py` (a routing-aware `PollLoop` `MessageHandler`). New commands:
  - **`cgc connect [NAME] [--space SPACE_ID] [--dispatcher]`** — create or reuse a session and open its **primary thread** via the existing threaded send (thread key = the session name), recording the returned `thread.name`. `NAME` is explicit or derived deterministically from the git repo basename + branch + a short SHA-256 of the absolute cwd (two checkouts of the same repo/branch get distinct, stable names; same inputs → same name, so reconnect is idempotent and posts nothing new). The **first** session auto-becomes the **dispatcher** when none exists; `--dispatcher` forces it (demoting any other so there is at most one). Prints routing instructions. Requires `webhook_url` and a resolvable space.
  - **`cgc session list`** — print sessions, their threads, and which is the dispatcher.
  - **`cgc disconnect NAME [--notify]`** — remove a session (optionally posting a "session disconnected" note to its primary thread); if it was the dispatcher and others remain, promote a survivor.
  - **`cgc listen --session NAME`** — routing-aware listen reusing the existing `PollLoop` (dedup/high-water/resilience/idle-timeout) and the shared `require_trigger`/catch-all conversion (new `listener.text_to_message`, DRY): for each new HUMAN message with thread `T` and text — emit if `T` is one of `NAME`'s claimed threads; if `T` is unclaimed and the text starts with `NAME:` (case-insensitive, optional space), **claim** `T` (persist `{name:T}`) and emit with the prefix stripped; if `T` is unclaimed, this session is the dispatcher, and the text starts with no registered session name, post a "which session?" menu to `T` (not emitted as work); otherwise skip (a thread owned by another session). `ChatMessage` gains an optional `session_name` field carried in the envelope/JSON (`to_jsonl`/`format_message`/`parse_message`) alongside `thread_name`, so every routed event names both its session and thread. New config `sessions_file` (`CGC_SESSIONS_FILE`, default `<config_dir>/sessions.json`, `0600`) wired through `Config`/`ENV_OVERRIDES`/`show`/`get`/`set` and a `complete_session_name` shell completer. Documented in `README.md`, `docs/usage.md`, `docs/configuration.md`, and `docs/architecture.md`. Real injectable tests (no network/disk/sleep) cover connect idempotency + default-name derivation, list/disconnect, dispatcher election + promotion, and every routing branch.
- **Thread-routing primitives** — two new capabilities for posting into and reading specific Google Chat threads:
  - **Threaded send** — `cgc chat send` gains an optional `--thread-key <KEY>`, threaded through `chat.send_webhook(config, msg, thread_key=...)`. When set, the webhook URL gets the `messageReplyOption=REPLY_MESSAGE_FALLBACK_TO_NEW_THREAD` query param and the JSON body gets `thread.threadKey`; repeated sends with the same key land in the same thread, an unseen key starts a new one. The created Message resource's stable `thread.name` is extracted from the webhook response and printed to **stderr** (`thread: spaces/.../threads/...`) so a caller can learn the thread.name for that key (needed for read-filtering); stdout stays the machine-readable `sent` line. With no `--thread-key` the behaviour and return (`None`) are unchanged. Webhook-secret redaction (query string stripped from error messages) and the non-2xx fail-fast are intact on the threaded path.
  - **Thread-filtered listen** — `cgc listen` gains a repeatable `--thread <THREAD_NAME>` option (config key `threads` / env `CGC_THREADS` as a comma list; a TOML array in `config.toml`). When one or more threads are configured the listener emits **only** messages whose raw `thread.name` is in that set, composed *in addition to* the existing sender-type (HUMAN-only) and `require_trigger` logic; with none configured the whole space is surfaced (unchanged). The owning `thread.name` is threaded through the listener predicate (new pure `rawmessage.thread_name` accessor) and surfaced on every emitted message: `ChatMessage` gains an optional `thread_name` field carried in the envelope/JSON (`to_jsonl` and `format_message`), preserving backward compatibility. Resilience, durable high-water state, and catch-all mode all continue to work with the thread filter. New config key `threads` is wired through `Config`/`ENV_OVERRIDES`, `config show`/`get`/`set` validation, and a `complete_thread` shell completer. Documented in `README.md`, `docs/usage.md`, `docs/configuration.md`, and `docs/architecture.md`.
- **Resilient poll loop** (`polling.py` + new `resilience.py`): each `cgc listen` / `cgc serve` poll iteration now survives **transient** backend errors — `socket.timeout`/`TimeoutError`, `ConnectionError`/`OSError`, `httplib2` transport errors, and Chat API `HttpError` with status in `{408,429,500,502,503,504}` — by writing a concise, secret-free diagnostic to stderr (exception type / HTTP status only; never a URL or response body) and continuing on the normal cadence instead of crashing. **Fatal** auth/permission errors (`HttpError` `401`/`403`) still fail fast immediately with an actionable message. A new config-driven bound `max_consecutive_errors` (`CGC_MAX_CONSECUTIVE_ERRORS`, default `10`, fail-fast int parsing) raises `PollLoopExhausted` and exits non-zero after that many *consecutive* transient failures (counter resets on any success), so a truly-down backend still surfaces. `run_to_exit_code` maps exhaustion to the shared non-zero exit alongside idle-timeout. Adds `httplib2>=0.22` as an explicit dependency.
- **Durable high-water state** (new `state.py`): a `StateStore` protocol with `FileStateStore` (JSON marker written `0600`; missing/corrupt file → fresh start, never a crash) and an injectable `InMemoryStateStore` test double. New config `state_file` (`CGC_STATE_FILE`, default `<config_dir>/listen-state.json`). `PollLoop` loads the last-processed `createTime` marker on startup and persists it on each advance, so a **restart resumes** from the last-processed message instead of re-reading recent history and re-emitting already-seen messages. `listener.run`/`serve.run` build a `FileStateStore` from `state_file`; the store is injectable so unit tests touch no real disk.
- **Catch-all listener mode** — new config `require_trigger` (`CGC_REQUIRE_TRIGGER`, default `true` = unchanged behavior, fail-fast boolean parsing). When `false`, `cgc listen` surfaces **every** message from a HUMAN sender regardless of the trigger prefix; non-human senders (BOT/app/webhook) are always excluded to prevent loops (the listener never echoes its own outbound posts or other bots). Trigger-prefixed human lines still parse as structured commands; plain conversational lines are surfaced via the new `messages.message_from_human_text` (a `command`-kind `ChatMessage` carrying the full text, which never fails fast unlike `parse_message`). Sender `type`/`email`/`thread` access is centralized in a new pure `rawmessage.py` module shared by `listener.py` and `serve.py` (DRY; replaces `serve.py`'s former private `_is_app_message`/`_message_sender_email`/`_thread_key` helpers).
- **Shared poll primitive** (`polling.py`): `PollLoop` holds the `_seen`/`_since` dedup, `createTime` high-water tracking, idle-timeout run loop, and one-JSON-line-per-message stdout emit shared by `listener.py` and `serve.py` (removes the duplicated bookkeeping and timeout/run wrappers). `run_to_exit_code` centralizes the idle-timeout → stderr → non-zero-exit mapping.
- **Shared validators** (`validation.py`): `validate_space_id` (the `spaces/<id>` rule, previously duplicated across `chat.py` and `bootstrap.py`) and `validate_create_time` (a new RFC3339 guard for the Chat list `createTime` filter, validated before interpolation).
- **`messages.to_jsonl`**: single canonical JSON-line serializer for stdout/log output, built from the same envelope as `format_message` (replaces divergent `json.dumps(asdict(msg))` paths in the listen/serve loops so wire and log shapes cannot drift).
- **Config-driven `webhook_timeout`** (`CGC_WEBHOOK_TIMEOUT`, default `30.0`) and **`page_size`** (`CGC_PAGE_SIZE`, default `100`), replacing the previously hardcoded webhook HTTP timeout and Chat list page size. Documented in `docs/configuration.md`.
- **Opt-in JSON envelope for outbound Chat text** — new `send_envelope` config key (`CGC_SEND_ENVELOPE`, default `false`, fail-fast boolean parsing) and a tri-state `cgc chat send --envelope/--no-envelope` flag (omit to use config; either flag overrides for that send). `format_message` gains a keyword-only `include_envelope` parameter; `chat.send_webhook` and `chat.post_message_as_app` pass `include_envelope=config.send_envelope`. Documented in `README.md`, `docs/configuration.md`, `docs/usage.md`, and `docs/architecture.md`.
- **`bootstrap.SpaceNotFoundError`**: a configured-but-nonexistent/inaccessible space id (HTTP 404) now surfaces a distinct, actionable "space not found or app lacks access" error instead of the misleading Chat-app-configuration gate.
- **Terraform inputs** `subscription_ack_deadline_seconds`, `subscription_message_retention_duration`, and `chat_push_service_account` (with validation and documented defaults), replacing hardcoded Pub/Sub tunables and the publisher service-account literal. Documented in `terraform/README.md` and `terraform.tfvars.example`.
- **Robust test suite (>90% coverage)**: unit tests for every module (`auth`, `bootstrap`, `chat`, `cli`, `config`, `listener`, `messages`, `serve`), plus `tests/integration/` flow tests and `tests/journeys/` end-to-end user-journey tests. Coverage is enforced in CI via `pytest --cov` with `--cov-fail-under=90` wired into `[tool.pytest.ini_options]`; the suite currently reports 100% line coverage. Adds the `freezegun`, `pytest-cov`, and `responses` dev dependencies (in `[dependency-groups].dev`) and ignores coverage artifacts (`.coverage`, `coverage.xml`, `htmlcov/`) in `.gitignore`.
- **Automated tag release pipeline** (`.github/workflows/release.yml`): on push to `main`, re-validates and reads the version from `pyproject.toml` (single source of truth), then cuts an annotated git tag `v<version>` plus a GitHub Release carrying the built `dist/*` artifacts. Idempotent — skips cleanly when the tag already exists, so it cuts no new release without a version bump.
- **PyPI publish workflow** (`.github/workflows/publish.yml`) using `pypa/gh-action-pypi-publish` with **OIDC Trusted Publishing** (no stored token) by default, plus a documented API-token fallback. Triggers only on a published GitHub Release or manual dispatch — never on the merge-validation path — and runs in a GitHub Environment named `pypi`. Inert until the maintainer completes one of the two PyPI setup options documented in `docs/installation.md`.
- **Makefile `publish` target** for validated manual publishing: `uv build` + `uvx twine check` + `uv publish`, reading the token from `UV_PUBLISH_TOKEN` (never hardcoded). Added `format-check` and `distcheck` targets.
- **`docs/usage.md`**: "Phone notifications (avoid duplicate alerts)" section covering the standalone Google Chat app vs. Gmail Chat surfaces and how to disable one to avoid duplicate phone alerts.
- **Secret-scanning CI** (`.github/workflows/gitleaks.yml`): runs the open-source gitleaks **binary** (pinned upstream release, no licensed action) on every push and pull request via `gitleaks detect --source . --redact --exit-code 1`, so no license or stored secret is required for the public repo.
- **gitleaks configuration** (`.gitleaks.toml`): extends the default ruleset and allowlists the project's known-safe placeholders (`SECRETKEY`, `SECRETTOKEN`, `spaces/AAAA`, `key=...&token=...`, `/tmp/client_secret.json`) and `tests/data/*.toml` fixtures so real secrets are still caught without false positives.
- **Pre-commit secret scan** (`.pre-commit-config.yaml`): a local gitleaks hook mirroring the CI gate; `CONTRIBUTING.md` documents `pip install pre-commit && pre-commit install` and notes that CI enforces the same scan.
- **`.gitignore` hardening**: appended de-duped ignore patterns for credentials and local state (`config.toml`/`*config.toml`, `*.key`, `*_key.json`/`*-key.json`, `client_secret*.json`, `credentials*.json`, `service-account*.json`/`sa*.json`, `*.pem`, `.env.*`, `*.tfstate`/`*.tfstate.*`, `.terraform/`) so no secret or state can be committed, while keeping `terraform.tfvars.example` tracked.
- **README "NOT READY" banner**: a prominent warning at the very top of `README.md` stating the project is experimental and must not be installed or relied on, and will be rewritten with a stable `v1.0.0` when ready.

### Changed

- **Clean human Chat messages by default**: outbound messages (`cgc chat send` and `cgc serve` replies) now post only the emoji-prefixed summary line — the JSON envelope is no longer embedded in the human-facing Chat view unless `send_envelope` is enabled (or `cgc chat send --envelope`). The machine-readable channel remains the JSONL on `cgc listen` / `cgc serve` stdout. Fixes raw JSON leaking into the human Chat view.
- **Default trigger prefix shortened to `claude:`** (`DEFAULT_TRIGGER_PREFIX` in `messages.py`, replacing the previous longer prefix). All docs, slash-command/skill markdown, the Terraform `trigger_prefix` default, and tests depending on the default were updated; `parse_message` still accepts both trigger-prefixed lines and fenced/bare JSON envelopes. Custom `trigger_prefix` values are unaffected.
- **Narrowed Chat-app not-configured classification** (`bootstrap.py`): the configuration gate (`ChatAppNotConfiguredError`) now fires on HTTP 403 `PERMISSION_DENIED` (and the explicit "is not configured" phrasing) only; a 404 on a configured space id maps to `SpaceNotFoundError` so operators get the correct remediation.
- **Truthful idempotent subscription result**: on HTTP 409 (subscription already exists), `bootstrap` now fetches and returns the real subscription resource name via `subscriptions.list` instead of a synthetic placeholder.
- **DRY refactors**: shared pagination and service-builder helpers in `chat.py`, a single `Config.require_keys` path for missing-value errors, shared `merge_config_values` validation for `cgc config set`, and a shared `_apply_overrides` CLI helper for `serve`/`listen`/`clear`.
- **Extracted `APP_MEMBER_NAME = "users/app"`** constant (with a comment noting it is the Chat API's fixed self-reference), replacing the inline literal.
- **`make lint` now also runs `ruff format --check`** (via the existing `format-check` target) so formatting drift is caught locally before push, matching the CI gate.
- **Removed prohibited suppressions** from the test suite (`# type: ignore[no-untyped-def]`, `# type: ignore[arg-type]`, `# pragma: no cover`) by fixing the root cause (typing the inner test function, using `typing.cast` for the deliberate invalid-input path, and refactoring the console-script lookup into a unit-tested helper).
- **Removed dead code**: the unused `complete_config_value` completer and its `_CONFIG_VALUE_KEYS` constant (never wired into the CLI) and their tests.
- **Pinned-action bump**: upgraded `actions/checkout@v4 -> @v5` and `astral-sh/setup-uv@v6 -> @v8` across the CI, release, and publish workflows to run on the current Node 24 runtime and clear the Node 20 deprecation warning.
- **CI** (`.github/workflows/ci.yml`) now includes a version-consistency gate (`pyproject.toml` vs `src/claude_google_chat/__init__.py:__version__`) and routes format checking through the `make format-check` target.
- **`docs/installation.md`** and **`docs/architecture.md`** document the build/release/publish pipeline, how to cut a release, how to finish PyPI setup (Trusted Publisher or API token), and the manual-publish path. `docs/installation.md` also notes the `cgc` console-command name overlap with an unrelated PyPI package and the unambiguous `python -m claude_google_chat` invocation.
- **Public documentation cleanup**: corrected the `docs/SETUP.md` runbook to match the Terraform module (service-account credentials via ADC/impersonation rather than a non-existent key-file output, the full set of enabled APIs, the Pub/Sub topic that `cgc bootstrap` requires, and the Pub/Sub connection step) and the rendered `config.toml` template (only valid `Config` keys); aligned the `cgc` command surface, install instructions, and `chat send` output wording across the README and docs; and added a "Two ways to run this" overview reconciling the webhook/OAuth and service-account auth paths.

## [0.1.0] - 2026-06-19

### Added

- **Claude Code plugin** installable from a marketplace (`/plugin marketplace add matthew-dresden/claude-google-chat`, `/plugin install claude-google-chat@claude-google-chat`), with the `.claude-plugin/plugin.json` and `marketplace.json` manifests.
- **Slash commands:**
  - `/claude-google-chat:chat-setup` — interactive configuration helper (webhook URL, space, OAuth, trigger prefix).
  - `/claude-google-chat:chat-send` — send a structured status ping to the configured Google Chat space via the incoming webhook.
  - `/claude-google-chat:chat-listener` — start the event-driven inbound listener and surface trigger-prefixed commands.
- **`google-chat` skill** documenting the ChatOps protocol: the structured message envelope, the on-the-wire representation, the inbound command trigger prefix, and the operational rules.
- **`Stop` hook** (`hooks/hooks.json`) that sends a status ping to the configured Google Chat space when a Claude Code session stops.
- **Python CLI `cgc`** (Typer) with subcommands `config init|show|get|set`, `auth login`, `chat send`, and `listen`, plus `--version`. (Later releases add `bootstrap`, `serve`, `clear`, `status`, `setup`, and `completion`.)
- **Structured message format** (`messages.py`): frozen `ChatMessage` dataclass with pure `format_message` / `parse_message` functions, strict validation of `version`/`kind`/`status`, and a single status→emoji map.
- **Env-first configuration** (`config.py`): file + environment merge with documented precedence, fail-fast on missing required values, defaults for non-secret tunables, OS config-dir storage via `platformdirs`, and secret-masking for display.
- **Google OAuth** (`auth.py`) via InstalledAppFlow with locally cached tokens written with restrictive permissions, used for inbound read/listen.
- **Google Chat integration** (`chat.py`): outbound send via incoming webhook, inbound list/delete via the Chat REST API.
- **Event-driven listener** (`listener.py`): env-driven poll cadence and idle timeout (no `sleep`-based readiness), `--once` drain mode, JSON-line output to stdout.
- **Documentation:** README plus `docs/installation.md`, `docs/usage.md`, `docs/configuration.md`, and `docs/architecture.md`.
- **CI and release workflows:** GitHub Actions CI (lint, format check, type check, test, build, manifest validation across Python 3.11 and 3.12) and an idempotent release workflow that tags `v<version>` and publishes built artifacts.
